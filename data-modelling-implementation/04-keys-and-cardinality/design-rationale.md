# Keys and Cardinality — Design Rationale

Reference for `01-keys-and-cardinality.sql`. Each section expands on the
engineering reasoning, interview talking points, and common traps.

---

## 1. Natural vs Surrogate Key Trade-offs

| Dimension | Natural Key | Surrogate Key |
|---|---|---|
| Source | Comes from the OLTP application | Generated by the warehouse |
| Meaning | Human-readable (emp_id = 22 means something to HR) | Meaningless integer (just a row locator) |
| Stability | Can change — a vendor rekeys their system, a merger renumbers employees | Never changes once assigned |
| SCD2 support | Cannot uniquely identify a historical version | One SK per version — designed for SCD2 |
| Join performance | VARCHAR or composite → slower index lookup | INT → smallest, fastest join column |
| Debuggability | Easy to trace back to source record | Requires a SK-to-natural-key lookup |

**Rule of thumb:** always use surrogate keys in the warehouse for dimensions
that participate in SCD Type 2, or that are joined to fact tables. Preserve
the natural key as a non-PK column for traceability and reconciliation.

### When natural keys are acceptable in the warehouse
- Very small, static reference tables (country codes, currency codes) where
  the natural key IS the surrogate (ISO-3166 `country_code = 'US'` is
  stable and short enough to join on directly).
- Degenerate dimensions: the natural key is stored directly in the fact
  table because no separate dimension table is needed (e.g.,
  `leave_request_id` in `fact_leave_lifecycle`).

## 2. Why Surrogate Keys Are Required for SCD Type 2

SCD Type 2 stores one row per version of a dimension member. When
employee 5 is promoted, the dim table looks like this:

| employee_sk | emp_id | job_title | effective_date | expiry_date | is_current |
|---|---|---|---|---|---|
| 101 | 5 | Data Analyst | 2020-03-01 | 2022-11-30 | 0 |
| 247 | 5 | Senior Analyst | 2022-12-01 | NULL | 1 |

Both rows have `emp_id = 5`. If the fact table stored `emp_id` instead of
`employee_sk`, there would be no way to know which version of employee 5
was active when a specific fact occurred.

By storing `employee_sk = 101` on a fact row from 2021, the query
automatically joins to the "Data Analyst" version of employee 5 — the
correct context for 2021 reporting — without any date-range filter on the
fact query itself.

**The surrogate key is what makes SCD2 reporting correct and efficient.**

### The lookup pattern
```sql
-- Correct: always join on the surrogate key
SELECT f.salary_amount, de.job_title
FROM fact_salary_payment f
JOIN dim_employee de ON de.employee_sk = f.employee_sk;
-- de.job_title is automatically the version that was current
-- when the payment was made.
```

```sql
-- Wrong: joining on the natural key without a date filter
SELECT f.salary_amount, de.job_title
FROM fact_salary_payment f
JOIN dim_employee de ON de.emp_id = f.emp_id AND de.is_current = 1;
-- This always shows TODAY's job title, even for historical payments.
-- 2021 payments show "Senior Analyst" even though the title was
-- "Data Analyst" at the time. Silently wrong.
```

## 3. Cardinality Types and Their DDL Implications

### 1:1 — One-to-one
**DDL implication:** add a UNIQUE constraint on the FK column in the child
table. Without the UNIQUE constraint, the database allows multiple child rows
per parent row, silently turning 1:1 into 1:M.

```sql
-- Example: one performance review per employee per period
ALTER TABLE performance_reviews
    ADD UNIQUE KEY uq_emp_period (emp_id, period);
```

Consider merging 1:1 tables unless there is a performance reason (the second
table is rarely queried), an access-control reason (the second table holds
PII), or a source-system reason (they come from different upstream systems).

### 1:M — One-to-many
**DDL implication:** the FK lives on the "many" side. Index the FK column
for join performance.

```sql
-- employees.dept_id is the FK; departments.dept_id is the PK
ALTER TABLE employees ADD INDEX idx_dept_id (dept_id);
```

### M:N — Many-to-many
**DDL implication:** you must create a junction/bridge table. Never store
a comma-separated list of IDs in a single column — that is a 1NF violation
and makes every query that uses those IDs a string-parsing problem.

```sql
-- project_assignments is the junction table for employees ↔ projects
-- It holds (emp_id, project_id) as a composite key
ALTER TABLE project_assignments
    ADD UNIQUE KEY uq_emp_project (emp_id, project_id);
```

In the warehouse, the equivalent is the bridge table `bridge_emp_project`,
which adds `allocation_weight` to prevent double-counting in semi-additive
measures.

## 4. The Surrogate Key Pipeline

The pipeline is a sequential lookup chain that converts natural keys (from
the OLTP) into surrogate keys (for the warehouse fact table) at load time.

```
OLTP source row
    │
    ├── emp_id        → JOIN dim_employee   WHERE is_current = 1  → employee_sk
    ├── dept_id       → JOIN dim_department ON dept_id            → dept_sk
    ├── job_title     → JOIN dim_job        ON job_title          → job_sk
    └── flag combo    → JOIN dim_employee_flags ON all flags      → flag_sk
                                                                      │
                                                            INSERT into fact table
```

**Critical detail:** the `is_current = 1` filter on `dim_employee` returns
the surrogate key for the CURRENT version of the employee. For historical
backfills, replace `is_current = 1` with a date-range filter:

```sql
AND pay_date BETWEEN de.effective_date
                 AND COALESCE(de.expiry_date, '9999-12-31')
```

This ensures a fact row loaded retroactively for a past pay date gets the
dimension version that was active at that time, not today's version.

### What happens when the lookup fails (no matching dim row)?

A failed JOIN drops the fact row silently. To detect this:
1. Use a LEFT JOIN and check for NULL surrogate keys before inserting.
2. Log the unmapped natural keys to an error table.
3. Fail the pipeline if the unmapped count exceeds a threshold (e.g., > 1%).

## 5. Interview Trap: "Why Not Just Use the Natural Key in the Fact Table?"

This is one of the most common interview questions in data modelling rounds.

**The trap:** it sounds simpler — why add an extra lookup step just to get
an integer when you already have emp_id?

**The correct answer:**

Storing `emp_id` in the fact table works only if `dim_employee` is SCD Type 0
or SCD Type 1 (no history rows). The moment you introduce SCD Type 2:

1. There are multiple rows in `dim_employee` for the same `emp_id`.
2. Joining `fact ON dim WHERE emp_id = fact.emp_id` returns multiple rows
   per fact (fan-out), inflating all aggregates silently.
3. To fix the fan-out you add `AND is_current = 1` to the join — but now
   every historical fact row shows today's job title / department, not the
   one that was true when the fact occurred.

**The surrogate key is the mechanism that makes historical accuracy free at
query time.** The cost is paid once, at load time (the SK lookup). Every
subsequent analytical query on the fact table just joins on an integer and
gets the right history automatically.

**Follow-up: "What if the dimension is SCD Type 1 only — still need surrogates?"**

Yes, for these reasons:
- The source system might change the natural key (rekey, merger). A surrogate
  insulates the warehouse from that event.
- Natural keys are often VARCHAR or composite, which are larger and slower
  to join than a 4-byte INT.
- The warehouse is built to be independent of source system decisions. Leaking
  source PKs into the fact table couples the two tightly.

The only acceptable exception is a degenerate dimension: a natural key
stored directly in the fact table when no separate dimension table exists
(e.g., `leave_request_id`, `order_number`). Those are natural keys on purpose
because they carry no attributes that would justify a dim table.
